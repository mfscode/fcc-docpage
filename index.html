GIT

This documentation is going to teach you on what, why, and how Git is

Quick Overview

So, what is git?

Imagine you're making a documentation about an animal, you've researched about that animal and now am ready to write the report.

After some writing, you research again about that animal and found out that your data is actually incorrect, so you delete the wrong part of your report and write a more accurate one.

Finally your report is ready, now you submit your report to your teacher, your teacher then store that report in their files. And since you made a copy of your report, you also saved one in your computer just in case the teacher found some inaccuracy in your report and ask you to fix it.

That's Git

It helps you to save a version of a file (or in this example, the report in a certain point of time), revert to the previous version of that file (e.g when the change of the file turns out breaking the file, you would like to revert to the version before the change), and then publish / store that file to a directory or, what we developer love to call, a repository, whether it's remote (on another side e.g GitHub) or locally in your storage system.

A (kinda) Short History Of Git

As with many great things in life, Git began with a bit of creative destruction and fiery controversy.

The Linux kernel is an open source software project of fairly large scope. During this early years of the Linux kernel maintenance (1991 - 2002), changes to the software were passed around as patches and archived files. In 2002, hte Linux kernel projecy began using a proprietary DVCS (Distributed Version Control Systems) called Bitkeeper

In 2005, the relationship between the community that developed the Linux kernel and the commercial company that developed Bitkeeper broke down, and the tool's free-of-Charge status was revoked. This prompted the Linux development community (and in particular Linus Torvalds, the creator of Linux) to develop their own tool based on some of the lessons they learned while using Bitkeeper. some of the goals of the new system were as follows:

  • Speed
  • Simple design
  • Strong support for non-linear development (thousands of parallel branches)
  • Fully distributed
  • Able to handle large projects like the Linux kernel efficiently (speed and data size)

Since its birth in 2005, Git has evolved and matured to be easy to use and yet retain these initial qualities. It's amazingly fast, it's very efficient with large projects, and it has an incredible branching system fir nin-linear development (See Git Branching)

Git Basics

This chapter covers every basic command like how to initialize a repo, how to clone a repo, adding and committing, and push it to a remote repo like GitHub

Getting a Git repository

You typically obtain Git repositories in one of two ways:

  1. You can take a local directory that is currently not under version control, and then turn it into a Git repository, or
  2. You can clone an existing Git repository from else where (either from a remote repo like from GitHub, or from other machines)

Initialize from local directory

If you have a project directory that is currently not under version control and you want to start controlling it with Git, you first need to go to that project's dirextory. If you've never done this, it looks like this (if you're on a windows system)

$ cd C:/Users/user/yourProjectName

And then type this:

$ git init

This creates a new subdirectory named .git that contains all of your necessary repository files — a Git repository skeleton. At this point, nothing in your project is tracked yet.

If your directory is not empty (you already have something in it e.g html files), after you initialize git in the folder, you would want to commit

$ git commit -m "yourMassageAboutTheCommit"

Clone an Existing Repository

If you want to get a copy of an existing Git repository -- for example, a project you'd like to contribute to-- the command you need is $ git clone -theRepositoryUrl-, The url doesn't need the dash (-) on it

For example, if you want to clone the Git linkable library called libgit2, you can do so like this:

$ git clone https://github.com/libgit2/libgit2

That creates a directory named libgit2, initializes a .git directory inside it, pulls down all the data for that repository, and checks out a working copy of the latest version. If you go into the new libgit2 directory that was just created, you'll see the project files in there, ready to be worked on or used

If you want to clone the repository into a directory named something other than libgit2, you can specify the new directory name as an additional argument:

$ git clone https://github.com/libgit2/libgit2 mylibgit

That command does the same thing as the previous one, but the target directory is called mylibgit.
